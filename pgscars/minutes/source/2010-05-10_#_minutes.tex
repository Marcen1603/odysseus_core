\subsection{Sensordatenfusion}
\begin{itemize}
	\item Pro Sensor gibt es eine Quelle
	\item Die Sensordatenfusion müssen wir selbst (in Odysseus) durchführen
	\item Das Schema des Kontextmodells (Attribute des Umweltmodells) müssen wir vorher selbst definieren
	\item Die Verknüpfung von Attributen aus verschiedenen Quellen erfolgt durch den (Kalman-)Filter
	\item Das Zusammenführen von neuen Objekten mit dem Umweltmodell erfolgt in der Assoziation  (siehe Abbildung \ref{fig1})
	\item Sensordatenfusion erfolgt anhand einer Schnittmenge (wertbasiert, keine ID!) (siehe Abbildung \ref{fig2})
	\item Beispielhafter Ablauf der Fusion (zu Abbildung \ref{fig2}):
	\begin{itemize}
	  \item Sensor 1 wird aktualisiert
	  \item \textit{A}, \textit{B} und \textit{C} werden mit neuen Werten von Sensor 1 belegt
	  \item \textit{C}, \textit{D} und \textit{E} werden prediziert
	  \item \textit{C} wird zum Assoziieren der neuen Messwerte mit den alten (prezidierten) Werten genutzt
	  \item Der Filter liefert alle Tupel \textit{A}, \textit{B}, \textit{C}, \textit{D}, \textit{E}
	  \item Ablauf wiederholt sich mit neuen Messwerten aus Sensor 2
	\end{itemize}
	\item Idealerweise liefern Sensor 1 und Sensor 2 abwechselnd neue Daten
	\item Die Fusion kann nur anhand der Schnittmenge erfolgen
	\item Die Schnittmenge muss eindeutig identifizieren können (Beispiel: Position, Gegenbeispiel: Farbe)
	\item Die Zusammenführung erfolgt im Broker
\end{itemize}

\subsection{Objektrelationalität}
\begin{itemize}
  \item Die Daten werden objektrelational gespeichert, d.h. in Form von ineinander verschachtelten Tupeln
  \item Programmiertechnisch wird dies durch ineinander verschachtelte Object-Arrays realisiert
\end{itemize}

\section{Diskusion unseres (Initialisierungs-)Konzepts mit André}
\subsection{\textit{Gesichert-}Operator}
\begin{itemize}
	\item Kann auch als Anfrage (SELECT * FROM stream WHERE \textit{Prädikat}) formuliert werden
	\item Prädikat bezieht sich auf Metadatum der Bewertungsfunktion
	\item Ansatz sollte funktionieren, da veraltete Tupel automatisch entfernt werden
\end{itemize}

\subsection{\textit{Bewerten und Löschen-}Operator}
\begin{itemize}
	\item Jedes Tupel besitzt Kovarianzmatrix als Metadatum
	\item Die Varianz kann prinzipiell als Attribut des Tupels oder als an das Tupel angehängtes Metadatum realisiert werden
	\item \textit{Bewerten und Löschen} kann zumindest im Prototyp als Anfrage realisiert werden
	\item \textit{Löschen} kann im Broker erfolgen
	\begin{itemize}
		\item In \textit{Bewerten} Metadatum setzen
		\item Prädikat fürs Löschen im Broker (mit Bezug zum Metadatum)
		\item \textit{Bewerten-}Knoten gehört im Anfrageplan hinter die \textit{Vereinigung}
	\end{itemize}
\end{itemize}

\subsection{Anforderungen an den Broker}
\begin{itemize}
  \item Wir müssen interne IDs vergeben, um alte Objekte aus dem Broker (Kontextmodell) entfernen zu können
  \item Die IDs sind vom Broker abhängig
  \item Wir können Select-Anfragen an den Broker stellen, da dieser eine Quelle ist
  \item Nähere Informationen zum Broker und Beschreibungen zu dessen Funktionsweise finden sich in der Diplomarbeit von D. Geesen
\end{itemize}

\subsection{Anforderungen an die Architektur}
\begin{itemize}
  \item Es soll flexibel gehandbhabt werden können, ob mehrere Quellen in einem Zyklus oder in mehreren Zyklen verarbeitet werden
  \item Die Architektur soll modular sein, dass wir auch andere Anfragepläne aufbauen können (keine feste Struktur, unsere "`Architektur"' stellt nur einen möglichen Anfrageplan dar)
\end{itemize}

\subsection{Anforderungen an die PQL}
\begin{itemize}
  \item Wir müssen in PQL ausdrücken können, welche Prediktion, Assoziation, Filterung, etc. verwendet werden soll
  \item Einführung eines zusammengefassten "`PAF-Statements"' kann unter Umständen zum Projektende angedacht werden
\end{itemize}

\subsection{Matlab}
\begin{itemize}
  \item Es besteht die Möglichkeit für mathematische Operationen (z.B. Matrizenmultiplikationen) Matlab anzubinden
  \item Dieser Ansatz sollte zumindest für Testzwecke ausreichend schnell sein
  \item Allerdings sollten wir Matlab keine Wahrscheinlichkeiten berechnen lassen, da dies sehr langsam sein soll
\end{itemize}
