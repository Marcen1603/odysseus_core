Im Paket validator befinden sich Klassen, mit deren Hilfe sich die Automaten im Metamodell auf Fehler überprüfen lassen. Dies geschieht, indem eine Reihe von Annahmen, die für Automaten, die im Metamodell spezifiziert wurden, nachgeprüft werden. Zur Zeit werden die folgenden Invarianten vorrausgesetzt:
  1. Startzustand muss vorhanden sein
  2. Startzustand muss in der Liste der Zustände enthalten sein
  3. Liste der Zustände darf nicht leer sein
  4. Jede Transition muss einen Zielzustand haben
  5. Jeder Zustand muss eine Liste von Transitionen haben (die aber leer sein kann)
  6. Jeder Zielzustand einer Transition muss in der Liste der Zustände enthalten sein
  7. Für jeden Automaten muss ein Ausgabeschema definiert sein
  8. Jede Transition muss eine Bedingung haben
  9. Jede Transition muss eine Aktion haben
  10. Jede Transitionsbedingung muss einen Jep-Ausdruck haben
  11. Jeder Automat muss ein Symboltabellenschema haben
  12. Jeder Eintrag im Symboltabellenschema muss einen eventidentifier, einen attributidentifier und eine abstrakte operation haben
  13. Jeder Automat muss eine Matching-Strategie haben
  14. Jeder Automat muss mindestens einen erreichbaren Endzustand haben
  15. Alle Zustände der Zustandsmenge des Automaten sollten erreichbar sein
  16. Jede Transitionsbedingung muss ein nicht leeres Label haben
  17. Alle Variablen, die im JEP-Ausdruck der Transitionsbedingung enthalten sind, müssen der Namenskonvention für Variablen entsprechen
  18. Alle Variable, die im JEP-Ausdruck des Ausgabeschemas enthalten sind, müssen der Namenskonvention für Variablen entsprechen
  19. Jedes Element im Ausgabeschema muss ein nicht leeres Label haben
Wird gegen eine Invariante verstoßen, so wird ein entsprechendes Fehlerobjekt erzeugt. Es wird nicht angenommen, dass die Liste der Invarianten vollständig ist. Das heißt, dass ein Automat, in dem keine Fehler gefunden wurden, nicht zwingend fehlerfrei sein muss.

Bei der Validierung werden alle entdeckten Verstöße zunächst gesammelt und erst nach Abschluss der Validierung zurückgegeben. Somit ist es möglich, dem Nutzer eine möglichst genaue Fehleranalyse zu bieten.

Für jeden entdeckten Verstoß gegen eine der Invarianten wird ein entsprechendes Ausnahmeobjekt erzeugt. Dabei wird zwischen Fehlern und Warnungen unterschieden. Fehler sind Verstöße gegen die Invarianten, die eine Verarbeitung des Automaten durch den EPA unmöglich machen. Warnungen sind hingegen weniger gravierende Verstöße gegen Invarianten. Trotz ihres Auftretens sollte ein Automat dennoch verarbeitet werden können. Die Vertsöße gegen die Invarianten deuten dennoch darauf hin, das der spezifizierte Automat dennoch (semantische) fehlerhaft ist. So deuten beispielsweise nicht erreichbare Zustände daraufhin, dass die Spezifikation von einigen Transitionen vergessen wurde.

Im Paket validator sind die Validatorklasse sowie weitere abstrakte Klassen zur definition des Fehlermodells enthalten. Im Paket error befinden sich konkrete Implemntierungen von Fehlern, wohingegen das Paket warning alle konkreten Warnungen enthält.
